企业组织管控平台
需求规格说明书

一、项目概述

1.1 项目背景
大型集团需要管理多种口径的组织架构关系,不同视角下公司间的父子关系可能不同:
- **法人组织树**:按股权控制关系或合并报表口径组织
- **预算组织树**:按预算管理归口组织
- **管理组织树**:按实际经营管理汇报关系组织

目前这些关系分散在各部门的 Excel 或业务系统中,缺乏统一权威源、历史追溯能力和对外统一供数机制。

1.2 项目目标
建设一个集团级组织关系管控平台,实现:
- 三棵组织树的权威维护和统一供数
- 任意历史时点的组织形态可查询(时光机)
- 变更的规范化流程、审批和追溯
- 与源头系统的一致性对账
- 向下游系统的变更事件推送

1.3 数据说明
- **实体颗粒度**:仅管理到公司级别,不包含部门
- **实体来源**:公司实体由源头系统(客商主数据/HR系统)同步产生,本平台不直接新建实体,只维护实体之间的组织关系
- **MVP范围**:mock 7家公司进行原型验证

 二、功能模块

2.1 组织浏览

 2.1.1 核心功能
在一个页面同时展示三棵组织树(法人树、预算树、管理树),支持时光机查询任意历史/未来时点的组织形态。

 2.1.2 交互设计

**筛选器**
- 日期选择器:支持选择历史、当前、未来日期(精确到日)
- 实体搜索框:模糊搜索公司名称或编码

**三树并列展示**
- 页面上方并排显示三棵树
- 选择日期后,三棵树同步切换到该日期的有效状态
- 点击任一树的节点,三棵树同时高亮该公司在各树中的位置
- 每棵树独立支持展开/收起

**树的展开逻辑**
- **首屏默认**:显示根节点(集团总部)及其直接子节点(两级)
- **搜索集团总部**:展示根节点 + 直接子节点(两级)
- **搜索其他公司**:展示该节点的父节点 + 该节点 + 该节点的子节点(三级)
- **祖先路径显示**:当节点层级较深时,从根到当前节点父节点的祖先路径以"面包屑"或"折叠链"形式显示在树顶部,让用户看到节点在全树中的位置

**下方信息面板**
- 点击节点后,下方表格显示该实体的基础信息
- 字段:公司编码、公司全称、简称、法人代表、注册资本、成立日期、状态、数据来源系统等

 2.1.3 技术要点
- 三棵树需高效渲染(虚拟滚动,懒加载)
- 跨树高亮同步
- 日期选择后需重新查询该时点的组织快照

 2.2 组织调整

 2.2.1 功能范围
本模块负责组织关系的日常维护,包括:
- **新增关系**:新公司成立后挂入组织树
- **修改关系**:变更某公司在树中的父节点(移动、划转)
- **终止关系**:公司注销后从树中移除

**变更支持未来生效**:允许提交未来某天生效的组织调整,组织浏览可查询未来形态。

**限制**:只能修改组织关系(父子层级),不能修改公司实体本身的属性(名称、法人等)。实体属性问题通过"模块4 实体主数据-问题上报"提交工单。

 2.2.2 操作流程

**发起变更**
1. 选择树类型(法人/预算/管理)
2. 选择变更类型:
   - 新增:选择子公司,选择父节点
   - 移动:选择目标公司,选择新父节点
   - 移除:选择目标公司,设置关系终止日期
3. **必填:业务生效日期**(effective_date)
   - 支持选择未来日期(组织调整往往提前决策、择日生效)
   - 不使用系统提交时间,必须由用户显式填写业务意义上的生效日
4. 填写变更原因(必填)

**提交前校验与预览**
- 系统自动运行**规则校验引擎**(见模块6)
- 展示 **diff 视图**:变更前后的树形态对比
- 显示校验结果:
  - **硬规则失败**(HARD):拦截提交,必须修改
  - **软规则警告**(WARN):允许用户确认后继续提交
  - **提示信息**(INFO):仅提示,不阻断

**并发控制**
- 同一节点存在待审批变更时,不允许再对该节点提交新变更(锁定)
- 系统提示:"该公司当前有待审批变更(变更ID XXX),请等待审批完成后再操作"

**提交进入审批**
- 变更提交后状态为"待审批"
- 生成变更记录(包含变更内容、校验结果、diff 快照)

 2.2.3 待办审批

**审批界面**
- 审批人查看:
  - 变更申请人、申请时间、业务生效日期
  - 变更内容(文字描述 + diff 视图)
  - 提交时的校验结果
  - 变更原因
- 审批操作:通过 / 拒绝(必填拒绝理由)

**审批通过后的处理**
1. **再次运行校验引擎**(关键):因为从提交到审批可能隔数天,树的状态可能已变,必须在写库前再校验一次,确保此刻该变更仍合法
2. 如果再次校验失败:
   - 自动驳回,通知申请人"审批期间数据已变更,原变更不再合法,请重新提交"
3. 如果校验通过:
   - 写入 `org_relation` 表,生效日期为用户指定的 effective_date
   - 如果是移动操作,需同时结束旧关系(设置旧关系的 expire_date = 新关系的 effective_date - 1天)
   - 生成**变更事件**,推送给订阅方(见模块3)
   - 记录操作日志

**审批拒绝后的处理**
- 变更记录标记为"已拒绝",不写入 org_relation
- 不产生历史记录,不生成变更事件
- 通知申请人

 2.3 监控与对账

 2.3.1 变更事件列表

**功能**:展示所有已生效的组织变更事件

**筛选条件**
- 树类型:法人/预算/管理/全部
- 事件类型:新增关系/修改关系/终止关系
- 时间范围:生效日期范围
- 实体名称:模糊搜索

**列表字段**
- 事件ID、树类型、事件类型、影响的公司、变更前父节点、变更后父节点、生效日期、操作人、操作时间

 2.3.2 推送投递状态看板

**功能**:监控变更事件向下游系统的推送状态

**看板展示**
- 按订阅方分组统计:
  - 总推送数、成功数、失败数、待重试数
  - 成功率、平均投递时延
- 失败明细:
  - 事件ID、订阅方、推送时间、失败原因、重试次数
  - 操作按钮:**手动重推**

**手动重推**
- 单条重推:点击某条失败记录的"重推"按钮
- 批量重推:勾选多条失败记录后批量重推
- 重推后更新投递状态

**MVP阶段说明**:mock 数据演示交互,不对接真实消息队列
 2.3.3 对账报告

**功能**:定期(或手动)生成数据一致性对账报告,发现并修复数据质量问题

 对账维度

**1) 与源头系统对账(实体层)**
- 源头有、本地没有:源头新增公司未同步
- 本地有、源头没有:源头删除公司但本地仍引用(脏数据)
- 两边都有但信息不一致:名称、编码、状态对不上
- 临时实体超期未转正:temp_flag 的实体超过 N 天未等到源头数据

**2) 树自身结构对账(关系层)**
运行校验引擎(模块6)检查:
- 孤悬节点:在树中但回溯不到根
- 游离实体:内部公司在三棵树中都不存在
- 时态断裂/重叠:某节点的父子关系时间线有空洞或重叠
- 任意时点非法:某历史日期拼出的树违反结构规则(多根、成环)

**3) 三棵树之间对账(跨树)**
- 法人树已注销、但预算/管理树仍有效的公司
- 只在一棵树出现的内部公司(提醒性质,不一定是错误)

 报告界面

**报告列表页**
- 字段:报告日期、报告类型(月度/手动)、数据健康分、差异总数、生成时间、状态
- 支持导出历史报告(Excel/PDF)
- 趋势图:展示健康分和差异数的月度变化

**报告详情页(汇总)**
```
数据健康分:  87 / 100   ↓ 比上月 -5

┌─ 与源头一致性 ─────────────────┐
│ 实体总数         1,248          │
│ 一致             1,201  (96.2%) │
│ 源头有你没有        12  ⚠       │
│ 你有源头没有         8  ⚠       │
│ 信息不一致          19  ⚠       │
│ 临时实体超期         8  ⚠       │
└────────────────────────────────┘

┌─ 树结构健康 ───────────────────┐
│ 孤悬节点             2  ✗ 严重  │
│ 时态断裂/重叠        3  ✗ 严重  │
│ 游离实体             5  ⚠       │
└────────────────────────────────┘

┌─ 跨树差异(提醒) ──────────────┐
│ 注销未联动          4           │
│ 单树独有            15          │
└────────────────────────────────┘
```

**差异明细下钻**
点击任一差异类别,进入明细列表:
- 列表字段:公司编码、公司名称、差异类型、问题描述、发现日期、源头值、本地值、状态(待处理/已处理/已忽略)
- 每条差异的操作按钮:
  - **立即同步**:将源头值覆盖本地
  - **转问题工单**:生成一条问题工单,流转到模块4处理
  - **忽略并记录原因**:标记为"已确认可忽略",不再重复报告
  - **去修复**:跳转到组织调整页面手动修复

**手动触发对账**
- 按钮:选择对账日期、选择对账范围(全量/某棵树)、立即触发
- 大数据量时异步生成,前端轮询状态

**MVP阶段说明**
- mock 一份"源头快照数据",故意和本地制造几条差异(源头多一家、本地多一家、名称不一致),演示对账价值
- 对账逻辑走通,但源头快照用静态 mock 数据

 2.4 实体主数据

 2.4.1 查询实体

**功能**:查询集团所有公司实体的基础信息

**筛选条件**
- 公司名称/编码:模糊搜索
- 状态:正常/停用/注销/全部
- 数据来源:客商主数据/HR系统/手工录入(临时)

**列表字段**
- 公司编码、公司全称、简称、法人代表、注册资本、成立日期、状态、来源系统、最后同步时间

**详情页**
- 展示实体的完整属性
- 显示该实体在三棵树中的位置(调用组织浏览的"跨树高亮"能力)
- 关联的变更历史

 2.4.2 问题上报

**功能**:发现实体数据质量问题时,生成问题工单

**问题类型**(字典管理)
- 实体信息有误(名称、法人、注册资本等错误)
- 新公司未同步(源头已建,本系统未同步进来)
- 实体状态不一致(源头注销,本地仍正常)
- 其他

**提交表单**
- 选择问题公司(可选,如果是"新公司未同步"则不选)
- 问题类型(必选)
- 问题描述(必填)
- 附件(可选)

**提交后**
- 生成问题工单,状态为"待处理"
- 通知实体数据管理员

 2.4.3 问题工单管理

**工单列表**
- 筛选:工单状态(待处理/处理中/已解决/已关闭)、问题类型、提交时间
- 字段:工单编号、问题公司、问题类型、提交人、提交时间、状态、处理人

**工单详情与处理**
- 查看:问题描述、附件、提交人
- 操作:
  - 指派给某人
  - 添加处理备注
  - 变更状态(处理中/已解决/已关闭)
- 如果问题涉及源头数据,记录"已通知源头系统"并跟踪

**闭环**
- 对账报告里的差异可一键转成问题工单
- 工单解决后,下次对账该差异应自动消失

 2.5 系统设置

 2.5.1 用户权限

**功能**:控制用户对本平台的访问和操作权限

**权限维度**

**1) 按树类型控制**
- 某用户只能查看/维护法人树,不能访问预算树和管理树
- 可多选

**2) 按操作控制**
- 只读:只能查看组织浏览和变更历史
- 维护:可提交组织调整,但需审批
- 审批:可审批待办
- 工单处理:可处理实体主数据问题工单
- 系统管理:可管理权限、订阅、字典

**3) 按数据范围控制**(大集团场景,MVP阶段预留不实现)
- 某用户只能查看/管理某个子集团下的组织
- 全局管理员可查看全部

**界面**
- 用户列表:展示所有用户及其权限
- 新增/编辑用户:填写用户信息,配置三个维度的权限
- 权限模板(可选):预定义几种角色模板(只读用户、数据维护员、审批员、管理员)快速授权

 2.5.2 推送与订阅管理

**功能**:管理下游系统对组织变更事件的订阅

**订阅配置**

**新建订阅**
- 订阅方名称(如:合并报表系统、预算系统、数据仓库)
- 订阅树类型:法人/预算/管理,可多选
- 订阅事件类型:新增关系/修改关系/终止关系/全部
- 推送方式:
  - Webhook(HTTP 回调):填写目标 URL、认证方式(Token/签名)
  - 消息队列:选择 Kafka/RocketMQ,填写 topic、消费者组
- 推送策略:
  - 重试次数:失败后最多重试 N 次
  - 重试间隔:指数退避或固定间隔
  - 失败告警:连续失败达到阈值后通知管理员

**订阅列表**
- 字段:订阅方、订阅内容、推送方式、状态(启用/禁用)、创建时间
- 操作:编辑、启用/禁用、删除、测试推送

**消息队列配置**(MVP阶段界面占位,不接真实MQ)
- Kafka/RocketMQ 的连接信息
- Topic 映射:法人树变更 → topicA,预算树变更 → topicB
- 序列化格式:JSON/Avro

 2.5.3 字典管理

**功能**:配置系统运行所需的枚举和参数,支持不改代码即可调整规则和业务逻辑

**MVP阶段**:界面框架搭建,配置项预留待开发,以下为最终需管理的字典类别

**字典类别**

**1) 树类型字典**
- 编码:LEGAL / BUDGET / MANAGEMENT
- 名称:法人组织树 / 预算组织树 / 管理组织树
- 扩展字段:描述、是否启用、排序

**2) 校验规则配置**
- 规则编码:SINGLE_PARENT、NO_CYCLE、MAX_DEPTH 等
- 规则级别:HARD(硬规则,不通过拦截) / WARN(警告) / INFO(提示)
- 是否启用:开关
- 参数配置:
  - 最大层级深度(MAX_DEPTH):默认 10
  - 健康分权重:孤悬节点扣 10 分,信息不一致扣 1 分,可配置

**3) 实体相关枚举**
- 实体状态:正常/停用/注销
- 来源系统:客商主数据/HR系统/手工录入
- 问题工单类型:信息有误/新公司未同步/状态不一致/其他

**4) 差异类型字典**(对账报告用)
- 编码:MISSING_IN_LOCAL、EXTRA_IN_LOCAL、MISMATCH、ORPHAN、FLOATING 等
- 中文映射:源头有你没有、你有源头没有、信息不一致、孤悬节点、游离实体
- 严重级别:CRITICAL / WARN / INFO

**5) 变更事件类型**
- 新增关系 / 修改关系 / 终止关系

**界面**
- 按类别分组展示字典列表
- 支持新增、编辑、删除(某些系统级字典不允许删除,只能禁用)
- 批量导入/导出(Excel)

 2.5.4 操作日志

**功能**:记录用户在平台上的所有关键操作,用于审计和问题追溯

**记录内容**
- 谁:用户账号、姓名
- 何时:操作时间(精确到秒)
- 在哪个模块:组织浏览/组织调整/实体主数据/系统设置等
- 做了什么:具体操作类型(查询/新增/修改/删除/审批通过/审批拒绝等)
- 操作对象:公司名称/变更记录ID/工单ID 等
- 操作结果:成功/失败
- 操作前后数据快照(可选,高敏感操作才记录,如审批通过、权限变更)

**日志级别**
- 敏感操作(必记录):组织调整提交、审批、权限变更、订阅配置变更
- 一般操作(可选记录):查询、导出、查看详情

**查询界面**
- 筛选:时间范围、用户、模块、操作类型
- 列表字段:时间、用户、模块、操作、对象、结果
- 详情:展开某条日志,查看完整信息和数据快照(如有)

**日志保留策略**
- 保留时间:默认保留 1 年,可配置
- 归档:超过保留期的日志自动归档或清理

 2.6 规则校验引擎(底座能力)

**定位**:不是独立的前端菜单,而是后端共享服务,被多处调用并将结果嵌入到相关页面展示

 2.6.1 调用场景

1. **组织调整提交前**:跑硬规则,配合 diff 预览,不通过则拦截提交
2. **待办审批时**:审批人查看提交时的校验结果
3. **审批通过写库前**:再次校验,确保此刻数据仍合法(最关键)
4. **对账报告生成时**:全量跑一遍,检查树结构健康,结果归档到报告

 2.6.2 校验规则分类

**结构完整性规则(树的基本形态)**

| 规则名称 | 级别 | 说明 |
|---------|------|------|
| 单一父节点 | HARD | 每个节点在同一棵树内只能有一个父节点 |
| 唯一根节点 | HARD | 每棵树有且仅有一个根(集团总部) |
| 无环校验 | HARD | 不能出现 A→B→C→A 的循环;移动节点时,新父节点不能是该节点自己或其任何后代 |
| 无自引用 | HARD | 节点的父节点不能是它自己 |
| 无孤悬节点 | HARD | 除根节点外,每个节点都必须能通过父子链回溯到根 |
| 父节点必须存在且有效 | HARD | 指定的 parent_id 必须是真实存在且状态正常的实体 |

**时态一致性规则(加上生效日期维度)**

| 规则名称 | 级别 | 说明 |
|---------|------|------|
| 父存在期覆盖子存在期 | HARD | 子节点的生效期间必须落在父节点存在期间之内 |
| 时态无重叠 | HARD | 同一节点在同一棵树里,不能有两条时间上重叠的父子关系 |
| 任意历史时点的树完整性 | HARD | 任意一天,整棵树都必须是合法的树(单根、无环、无孤悬);变更提交前,对受影响的时间段做逐时点校验 |
| 生效日期不能早于实体成立日 | HARD | 关系的生效日不能早于公司实体的成立日期 |
| 未来生效不冲突 | WARN | 已录入未来生效变更的节点,再提交同日期的变更需提示冲突 |

**业务合理性规则(软规则,可开关)**

| 规则名称 | 级别 | 说明 |
|---------|------|------|
| 层级深度上限 | WARN | 超过 N 层(默认 10,可配置)提醒 |
| 父子实体类型合理性 | WARN | 不能出现"子公司下挂集团"的类型倒挂 |
| 孤子/独子提醒 | INFO | 某节点突然没有任何子节点,可能误操作 |
| 大批量变更预警 | WARN | 一次影响超过 N 个节点,提示二次确认 |

**跨树规则(提醒性质)**

| 规则名称 | 级别 | 说明 |
|---------|------|------|
| 实体一致性 | HARD | 出现在任一树里的公司,必须是实体主数据里存在的 |
| 停用/注销联动提醒 | WARN | 某公司在法人树已注销,但预算/管理树仍有效 |
| 游离实体检查 | WARN | 内部公司在三棵树里都找不到位置 |

 2.6.3 校验结果展示

**组织调整页面的校验面板**
- 位于 diff 视图旁边或下方
- 红色❌:硬规则失败,阻断提交,必须修改
- 黄色⚠:软规则警告,允许用户确认后继续
- 蓝色ℹ:提示信息,不阻断

**示例**
```
校验结果:
❌ 硬规则失败 (不可提交)
  - 无环校验:新父节点"华南分公司"是当前节点"集团总部"的子孙节点,会形成环

⚠ 警告 (可确认后提交)
  - 层级深度:移动后该节点将位于第 11 层,超过推荐上限 10 层

ℹ 提示
  - 该公司在预算树中仍挂在旧父节点下,建议同步调整
```

 2.6.4 核心算法说明

**无环检测算法**
```
从新父节点开始,沿父链向上回溯到根,
路径中如果出现被移动节点,则会成环,拒绝。
```

**任意时点树合法性校验(优化算法)**
```
不逐天遍历,而是收集受影响时间段内所有关系的生效日和失效日,
去重排序得到"关键时点"列表。
只在每个关键时点重建树快照并校验结构,性能可控。
```

 三、非功能需求

 3.1 性能要求
- 组织浏览:三棵树同时渲染 1000 节点时,首屏加载 < 2s
- 搜索响应:< 500ms
- 对账报告生成:大数据量时异步,< 5 分钟

 3.2 安全要求
- 用户认证:集成企业 SSO/AD
- 数据权限:按树类型、操作类型、数据范围三维控制
- 敏感操作审计:组织调整、权限变更、审批必记录操作日志

 3.3 可用性要求
- 系统可用性:99.5%(非核心交易系统)
- 数据备份:每日备份,保留 30 天

 3.4 扩展性要求
- 支持树类型扩展(未来可能增加"财务核算树"等)
- 支持校验规则插件化扩展
- 推送方式支

 四、数据模型(核心表)

 4.1 实体表 org_entity
公司实体主数据,从源头系统同步。

| 字段 | 类型 | 说明 |
|------|------|------|
| entity_id | VARCHAR(32) | 主键 |
| entity_code | VARCHAR(64) | 公司编码 |
| entity_name | VARCHAR(200) | 公司全称 |
| short_name | VARCHAR(100) | 简称 |
| legal_person | VARCHAR(100) | 法人代表 |
| registered_capital | DECIMAL | 注册资本 |
| establish_date | DATE | 成立日期 |
| status | VARCHAR(20) | 正常/停用/注销 |
| source_system | VARCHAR(40) | 来源系统 |
| source_ref_id | VARCHAR(64) | 源头系统的引用ID |
| temp_flag | TINYINT(1) | 是否临时实体 |
| synced_at | DATETIME | 最后同步时间 |

 4.2 组织关系表 org_relation
三棵树的父子关系,带时态。这是系统的核心表。

| 字段 | 类型 | 说明 |
|------|------|------|
| relation_id | BIGINT | 主键 |
| tree_type | VARCHAR(20) | 树类型 LEGAL/BUDGET/MANAGEMENT |
| entity_id | VARCHAR(32) | 子节点(当前公司) |
| parent_id | VARCHAR(32) | 父节点,根节点为 NULL |
| effective_date | DATE | 生效日期(业务生效日) |
| expire_date | DATE | 失效日期,当前有效为 9999-12-31 |
| created_by | VARCHAR(64) | 创建人 |
| created_at | DATETIME | 创建时间 |
| change_id | BIGINT | 关联的变更记录 |

**时态查询原理**:查某天的树 = `WHERE tree_type=? AND effective_date <= 查询日 AND expire_date >= 查询日`。移动节点 = 结束旧关系(expire_date=新生效日-1)+ 新增新关系。
